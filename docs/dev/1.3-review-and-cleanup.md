## 一、你发现的"资产页拖动排序消失" —— 真相：功能没删，是被有意改了交互

**结论：这是良性改动，不是回归，代码层面不需要修复。**

- 你以前在资产页可以**直接按住分组标题（现金/储蓄/信用卡这类分区）拖动排序**。
- 后来有一次提交（信息是"资产分区拖动改为显式排序模式，消除折叠错位和卡顿"，2026-07-03 提交，从 v1.1.3 版本开始生效，所以你装的 1.2.0 已经是新交互了）把它改成了：**先点【排序】按钮进入排序模式，分区自动折叠成标题并显示拖动手柄，拖完再点【完成】**。
- 为什么要改：旧的"直接按住就拖"实现，在分区展开/折叠时会出现手指和卡片错位、卡顿、第二个分区拖不动等问题。新的显式排序模式解决了这些。
- **没有受影响的部分**：
  - 分区内**单个账户行**的排序仍然是**长按直接拖**，一直可用。
  - 账户分组管理页里的分组排序照旧。
  - 排序结果的保存（存在本地、按账本隔离、进备份）完全没动。

**所以你感觉"消失"，实际是入口变隐蔽了**——分组排序现在藏在【排序】按钮后面，不像以前按住就能拖那么直觉。功能和数据都在。

如果你觉得这个【排序】入口不够显眼，我可以把它做得更醒目一点（比如放进资产页右上角的操作菜单，或者分区多的时候给个提示）。这属于**可选的体验优化，不是 bug**，做不做你定。

### 关于"校验所有版本看还有没有别的功能悄悄变了"

逐个版本翻代码 diff 效率太低。项目里本来就有一份记录"每个功能设计上应该是什么行为"的清单文件。更高效的做法是：拿那份清单和现在的实际代码逐条对一遍，凡是"清单说该有、现在没了或路径变了"的都标出来，判断是有意改的还是真丢了。这件事我建议单独做一次（列在下面的待办里）。目前为止，只发现拖动排序这一处感知差异，而且是良性的。

## 二、你问的：加了图片附件后，导出 JSON 会不会出问题？

不会，现在已经处理好了，方式是：

- 用户拍照或选相册图后，图片会先被压缩（最长边缩到 1600 像素、质量 80 的 JPEG），然后编码成一串 **base64 文本**（就是那种 `data:image/jpeg;base64,....` 的字符串）。
- 导出时，这串文本就当普通字符串塞进 JSON 里。导入时再还原成图片。**JSON 完全装得下图片，它只是变成了一段很长的文本。**
- 图片存在一张**独立的附件表**里，而不是和交易记录混在一起。这是特意的：如果和交易混在一起，每记一笔账都会把所有图片的 base64 重新写一遍，非常浪费；独立表只在增删图片时才写。

**唯一的代价**：base64 会让数据比原图大约 33%，再加上图片本身，**附件一多，备份文件就会变得很大**。如果以后这成了问题，可以把备份改成"一个 JSON + 一个图片文件夹，打包成 zip"的形式。这属于以后可选的优化，现在不急。

## 三、你的顾虑：数据库迁移脚本越来越多，会不会拖慢性能？

**不会。可以放心继续加迁移。** 原因：

- **全新安装 App**：直接一次性建好当前最新的表结构，那些历史迁移脚本**一个都不会跑**。零开销。
- **老手机升级 App**：只会跑"你手机上旧版本 → 新版本"这中间缺的那几步，而且**每一步一辈子只跑一次**，跑完就记下来了，下次启动不再跑。

所以迁移脚本积累再多，对启动速度和运行速度几乎没有影响——它不是每次开 App 都跑一遍的东西。

那"迁移多了"真正的代价是什么？是**代码变长、不好读**，不是性能。

你允许开发期直接改数据库结构，这个授权有用。有两种做法：

- **保持现在这样，继续加增量迁移**：好处是任何装过旧版本的手机都能平滑升级、不丢数据。在正式对外发布前，建议保持这样。
- **把历史迁移"压平"成一个干净的初始版本**：把 v2 到 v8 这些中间步骤合并成一个初始建表，删掉中间脚本。代价是任何跨过这个节点的旧手机升级需要重装（会丢数据）。因为现在实际上只有你一个人装 APK 测试，重装一次没影响。

我的建议：**现在不用压平**（纯粹为了代码整洁、不为性能），等第一次正式公开发布时再做一次干净的压平就好。

## 四、彻底移除 Web 端，只保留 Android

先说一个关键点，它决定了这件事的风险有多小：**跑测试时用的是 Android/io 那套代码，不是 Web 那套。** 所以把 Web 删掉，测试完全不受影响，全部会照常通过。

具体要动的东西：

**可以整个删掉的：**
- 整个 `web/` 文件夹（里面包括给浏览器用的 SQLite 文件 `sqlite3.wasm`、`sqflite_sw.js，Android 打包根本不读它们）。
- 6 个专为 Web 写的实现文件（文件名都以 `_web.dart` 结尾）：本地存储、数据库工厂、头像选择、数据导出、备份目录、WebDAV 客户端各一个。它们只被各自的入口引用，删掉没有连带影响。

**要删一行依赖：**
- 项目依赖清单（pubspec.yaml）里只需删掉一行：`sqflite_common_ffi_web`（这是 Web 版 SQLite）。
- **注意别删错**：还有两个名字很像的不能动——`sqflite_common_ffi`（测试要用）、`sqflite_common`（Android 要用）。就删带 `_web` 的那一个。

**要改的入口文件（每个删一行）：**
- 有 7 个"入口"文件，里面各有一行写着"如果是 Web 就用 Web 版"。把这一行删掉即可。删完之后，这些文件就变成"要么用 Android 版、要么用兜底版"，和项目里已经这样写的生物解锁、记账提醒两个模块长得一模一样，风格反而统一了。
- 那些"兜底版"文件（文件名以 `_stub.dart` 结尾）**建议本次保留不动**——它们在 Android 上不会被用到、也无害，删它们收益很小、还容易出岔子。

**改 GitHub 构建流程（这就是你说的"前面跑了个构建 Web 的多余步骤"）：**
- 构建配置文件 `.github/workflows/flutter.yml` 里，删掉"构建 Web"和"上传 Web 产物"这两步，并把那个步骤的名字里的"Web Build"去掉。
- 后面的"构建 Android APK"步骤依赖前面的"检查"步骤，删掉 Web 相关步骤后，"检查"步骤仍然会跑代码分析和测试，依赖链不会断，Android 照常构建。

**要同步改的文档（不改会留下误导信息）：**
- README.md、AGENTS.md、CLAUDE.md，以及 docs 文件夹下几个文件里，凡是提到"用 Chrome 预览""构建 Web""Web 端不支持 WebDAV/通知""三端适配"这类描述，都要改成只讲 Android 的说法。
- 特别是 WebDAV 和记账提醒，原来文档写"Web 端不支持"，删掉 Web 后它们在 Android 上一直可用，措辞要改成直接说 Android 的行为。
- 另外，删掉 Web 后本地预览就不能再用浏览器了，只能用 Android 模拟器或真机，开发说明也要跟着改。

## 五、其余隐性问题排查

整体扫了一遍，**项目很健康**：代码里没有遗留的"待修/临时凑合"标记，版本号在三个地方（依赖清单、版本文件、Git 标签）都一致（都是 v1.3.0+14），没有明显的技术债。补充几点记录：

1. **附件让备份变大**（上面第二节讲过）：图片以 base64 塞进 JSON，附件多了备份文件会很大。以后可选优化成 zip 打包。
2. **数据库迁移压平**（上面第三节讲过）：不影响性能，纯代码整洁问题，等正式发布时再做。
3. 有一个数据库工厂的"兜底"文件里写了一句报错代码，在 Android 和测试环境下永远走不到，无害，留着删掉都行。
4. 建议以后单独做一次**更深的结构审查**：看看那个装了全部应用状态的核心文件（veri_fin_controller.dart）是不是太大了、耦合是不是太重、能不能拆分。这个不紧急。
