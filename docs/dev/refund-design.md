## 0. 一句话背景

现在的「退款」只是挂在原支出上的一个**累计数字**，装不下「多笔、各有日期/账户/备注」。Issue #6 希望退款能记：**退款金额 / 备注 / 发起日期 / 到账日期 / 转入账户**，并支持**全额 / 部分 / 超额**。这需要把退款从「一个数」升级成「一等的、可见的事件」。

---

## 1. 名词解释（人话）

**场景**：6/28 你在淘宝买了个 100 块的东西，7/1 申请退货退款，但钱不会马上回来，**7/4 才真正到支付宝账上**。退款几乎都有这个时间差（1~7 天，信用卡有时更久）。

- **到账日期（7/4）** = 钱真正回到账户那天 → **账户余额、对账靠它**，刚需。
- **发起日期（7/1）** = 你点「申请退款」那天 → 纯信息，可有可无。
- **待退款追踪** = 一张「**别人还欠我、钱还没到**」的清单。价值两点：
  1. **不会忘**：网购退货多时常有「申请了、商家拖着没到账」，有清单能发现「这笔两周前的退款怎么还没到，该去催了」。
  2. **预算更准**：这 100 块要退你但还没到手，追踪它能让 App 提示「有钱在路上，先别当已有」。

  商业软件（Quicken Simplifi）就是这么做：先记「待到账退款」→ 显示「多少钱在等着回来」→ 到账了点「已核销」。

**我的倾向**：对「几乎不用退款」的场景，待退款追踪偏进阶。建议 **v1 先不做追踪、也先不要发起日期**，但**数据模型留好口子**（以后加个「是否到账」标志就能升级，不重新迁移数据）。这两点最终看你（见 §7 决策点）。

## 2. 业界怎么做（研究结论）

| 结论 | 出处 |
|---|---|
| **绝不删原交易**，退款记成「关联/冲抵」条目，保留历史与余额 | Wave / Hubifi |
| **退款算「负支出」，冲减原分类、不算收入**，让支出报表反映「你真正自己花掉的钱」 | Goodbudget |
| **部分退款 = 挂在原交易上的关联交易**，原单据仍附原交易 | Hubifi / Wave |
| **退款要能被看见**：显示成一条带「REFUND」标记、金额为正的交易，就在支出区里 | Quicken Simplifi |
| **退款要记到账账户 + 原因备注**；到账账户通常是原支付方式但可不同 | Wave / Hubifi |
| **待退款可先记「待到账」**，到账后核销闭环 | Quicken Simplifi |

出处链接见文末「参考」。

## 3. 现状（改造前）

- 退款不是独立条目，而是挂在原**支出**上的累计标量 `LedgerEntry.refundedAmount`（钳制 `[0, amount]`）。
- 净额 `netAmount = (amount − refundedAmount).clamp(0, amount)`；**余额与所有统计都按净额算**。
- 回款默认回**原账户**、**不产生独立收入条目**；**没有**日期 / 备注 / 转入账户 / 多笔 / 超额。
- 编辑入口：交易详情页 `transaction_detail_page.dart` 的「退款」行 → 数字键盘改一个数。
- 「待报销」`reimbursable` 只是一个筛选标记（不影响金额）。
- 一木账单导入的「退款」列也映射到这个标量。
- 用到净额/退款的地方：`ledger_math`、`home_metrics`、交易列表「报销状态」筛选（全部/待报销/已退）、`common_widgets` 交易行。

技术决策原记录见 `tech-decisions.md` 的「报销/退款模型」行（定案后需改写该行）。

## 4. 目标（改造后要满足）

1. 一笔支出可挂**多笔**退款，每笔各有：**金额 / 到账账户 / 到账日期 / 备注**。
2. 支持**全额 / 部分**退款；**超额**见决策点 §7.3。
3. 退款是**时间线上可见的一条**（账户流水里能看到「+¥XX 退款到 X」），不是偷偷改余额。
4. 退款**不计收入**、**冲减原支出所在分类**（净额口径不变）。
5. 到账账户可**不同于**原支付账户。
6. 老数据（现有 `refundedAmount`）平滑迁移；备份/导入/样例全覆盖。
7. 对后续维护友好：改动可控、有充分测试。

## 5. 方案定案建议：退款做成「关联条目」（记作方案 B1）

**核心思路**：退款 = 一条挂在原支出上的**独立 `LedgerEntry`**，类型为新增的 `EntryType.refund`，行为**类比转账**（不计收支、只动账户余额），额外多一条规则「冲减原支出净额」。好处：天然落在时间线与账户流水里，还能自带附件（退款截图）。

### 5.1 数据模型

新增 `EntryType.refund`；退款条目复用 `LedgerEntry` 字段，新增链接字段 `refundOf` 与到账日期 `settledAt`：

| 字段 | 含义 | 默认值 |
|---|---|---|
| `refundOf`（新增，`String?`） | 指向原支出 `id` | 必填（退款必须挂靠某支出） |
| `amount` | 本笔退款额 | 「剩余可退」= 原额 − 已退合计（含待到账） |
| `accountId` | **到账账户** | 原支出的 `accountId` |
| `occurredAt`（复用） | **发起日期**（申请退款那天，可回填） | 今天 |
| `settledAt`（新增，`DateTime?`） | **到账日期**；`null` = 待到账（pending） | 视录入而定 |
| `note` | 备注/原因 | 空 |
| `categoryId` | 冗余存原支出的分类（便于展示/兜底） | 原支出分类 |

> 发起日期复用现成的 `occurredAt`（一定有）；到账日期用新增可空的 `settledAt`（没到账就是 `null`）。「已到账」判定 = `settledAt != null`。

### 5.2 各口径怎么算（只认「已到账」的退款）

**关键**：下面所有金额只算 `settledAt != null`（已到账）的退款；**待到账退款不进任何金额**，只进「待退款」清单（原因见 §5.5）。

- **原支出净额**：`净额 = amount − Σ(挂它的·已到账·退款.amount)`，floor 到 0。`refundedAmount` 标量弃用（迁移见 §6）。
- **账户余额**：从今天的「原支出只扣净额」改成 **「原支出扣全额、每笔·已到账·退款给到账账户 +amount」**。
  - 到账账户 = 原账户时，结果和现在**完全一样**（等价净额）。
  - 到账账户 ≠ 原账户时，才体现差异（正好支持「转入账户」）。
- **收支统计**：退款条目**从收支聚合排除**（和转账一样）；支出分类排行按「原支出净额」算，退款本身不进支出和 → **不会重复计**。净额的削减落在**原支出所在的月份/分类**（沿用现有 `netAmount` 口径，即退款到账后原那笔支出的净额下降）。
- **首页/预算/小组件**：已是净额口径，自动正确，无需改。

### 5.3 生命周期规则

- 退款只能从「**原支出 → 添加退款**」入口创建；**普通记账页的类型选择器排除 `refund`**（不能手动把类型选成退款）。
- 删除原支出 → **级联删除**它的退款条目（仿附件级联）。
- 删除某笔退款 → 原支出净额自动恢复。
- 编辑原支出金额到低于「已退合计」→ 净额 floor 0（沿用现有钳制思路），并在 UI 提示。

### 5.4 低风险替代方案（B-hybrid，供对比）

保留 `refundedAmount` 标量当净额驱动（核心统计几乎不动），另建 `refunds` 明细表存「日期/账户/备注/多笔」。
- 优点：动核心数学最少、风险最低。
- 缺点：退款**不自动上时间线**（要另做展示）、且有「标量 = 明细之和」的**双源一致性**负担、附件无法挂到退款。

**我更推荐 B1**（可见性、商业化程度、长期干净都更好）。但如果你更怕动核心统计，B-hybrid 也能交付。

> 📝 **决策点 A — 选 B1 还是 B-hybrid**：「建议你做最好的就好，最好的应该就是 B1 吧？我理解的，你要做的退款就是可以点进去，然后看到一个类似于时间线的列表？并且可以附带附件之类的东西对吧？」

✅ **定案：走 B1**。你的理解正确——原支出详情页能点进「退款」区看到一条条退款（金额/账户/日期/状态/备注）的列表、可加多笔、每笔可带附件；已到账的退款还会作为一条正向交易出现在主时间线与账户流水里。

### 5.5 待退款（pending）生命周期与「为什么待到账不算钱」

退款有两态，由 `settledAt` 是否为空区分：

- **待到账（`settledAt == null`）**：你已申请、钱还在路上。**不进余额、不冲减净额、不进收支统计**，只出现在「待退款」清单里（可催、可核销）。
- **已到账（`settledAt != null`）**：钱真回来了。进到账账户余额、冲减原支出净额，并作为一条交易出现在时间线的到账日那天。

**录入流程**：加退款时选「已到账」（填到账日期，默认今天、可回填）或「待到账」（先只填发起日期）。之后在「待退款」清单点「已到账」补一个到账日期即可核销。

> ⚠️ **我替你定的口径（cash basis，请确认）**：待到账退款**不减少你的花费**。理由——钱还没真回来、账户余额也没动，此时把支出算少了会与余额对不上、也会让你误以为「已经省下了」；只有钱**真到账**才冲减花费。这与账户余额、与「预算别把在路上的钱当已有」都一致，也是 Simplifi 等的做法。若你更想「一申请就立刻减花费」，告诉我改。

## 6. 迁移 / 备份 / 导入

- **老数据迁移**：载入时自愈——对 `refundedAmount > 0` 的支出，合成一条退款条目（日期取原支出日、账户取原账户、备注「历史退款」），随后该支出的标量清零/弃用。此迁移**动钱**，需重点测试。
- **数据库**：提升 `AppDatabase.schemaVersion`，`entries` 表加 `refund_of` 列并写 `_onUpgrade` 迁移；`EntryType` 增加 `refund` 的序列化字符串。
- **备份 JSON**：退款条目随 `entries` 进出；`refundOf` 字段序列化。更新样例备份 `docs/dev/verifin-sample-backup.json`（加一条带退款的支出，测试能真实导入）。
- **一木导入**：原「退款」列改成——建原支出后再生成一条关联退款条目（而非写标量）。
- **导入预览**：退款条目在预览页的呈现（是否单列、是否可编辑）需一并设计。

## 7. 边界与决策点（请逐个批注）

### 7.1 待退款追踪（pending）做不做

见 §1 人话解释。我建议 **v1 不做，模型留口子**（以后给退款条目加 `settled`/`arrivedAt` 即可：未到账的退款不进余额/统计，只在「待退款」清单里显示）。

> 📝 **决策点 B**：你/用户网购退货是否频繁到需要「催退款清单」？v1 做还是不做？「这个 Issue 本来就不是我提出来的，本来就是为了满足别人的要求。有句话说得好：我可以不用，但是你不能没有。所以，还是最好直接做了，不要后面再缝缝补补，分批一次性弄好了，不是让我们的软件更好吗？」

✅ **定案：做**——含待到账/已到账两态、「待退款」清单与核销，一次做齐。

### 7.2 发起日期要不要

我建议 **v1 只留「到账日期」（可回填过去）**，发起日期先并进备注；等做「待退款追踪」时再引入。

> 📝 **决策点 C**：v1 只留到账日期够不够？「同上，最好直接一次性设计好，做好了，不要留到后面再补，那样就太麻烦了。」

✅ **定案：双日期都要**——发起日期（`occurredAt`）+ 到账日期（`settledAt`）。

### 7.3 超额退款（退回 > 原支出）怎么办

现实里存在（赔偿、多退少补）。三选一：
- **① 禁止**：最多退到全额（最简单、最稳）。
- **② 允许记录，但净额 floor 到 0**：超出部分不虚增收入，仅作记录。
- **③ 超出部分单独算一笔收入**：账更「平」，但引入退款↔收入的转换复杂度。

我倾向 **① 或 ②**。

> 📝 **决策点 D**：选哪个？「最多退款到全额，如果超出去了，提示一下用户之类的。」

✅ **定案：选 ①**——禁止超额，最多退到全额；用户输入会导致 Σ退款 > 原额时，提示并截到「剩余可退」。

### 7.4 报销与退款是否统一

现在「退款」和「报销回款」共用同一套字段。建议**继续统一**：退款条目同时服务报销场景，保留「待报销」标记作为筛选维度。

> 📝 **决策点 E**：认同统一吗？「认同」

✅ **定案：统一**——退款条目同时服务报销回款，保留「待报销」标记作为筛选维度。

## 8. 用户交互与展示（细节）

- **入口**：原支出详情页，把现在那行单数字「退款」换成「**退款**」区——列出每笔退款（金额/到账账户/状态[待到账·已到账]/日期/备注）+「添加退款」按钮，可加多笔；每笔可点开编辑、带附件。
- **添加/编辑退款弹窗**：金额（默认「剩余可退」）、到账账户（默认原账户、可改）、备注，以及**状态**——「已到账」填到账日期（默认今天、可回填），「待到账」只填发起日期。全额一键填满剩余；部分输金额；**想退超过剩余时提示并截到剩余**（决策 D）。
- **时间线/账户流水**：**已到账**退款以「**退款 +¥XX 到「支付宝」**」一条、正向色 +「退款」标显示（仿 Simplifi），落在到账日、点进去可跳回原支出；**待到账**退款不进时间线。
- **待退款清单**：一个可进入的列表（入口放交易页筛选/我的-数据工具，具体待定），汇总所有「待到账」退款、显示「合计 ¥XX 在路上」，每项可一键「标记已到账」。
- **原支出卡**：显示「已退 N 笔 · 已到账合计 ¥XX · 净支出 ¥YY（· 待到账 ¥ZZ）」。
- **报表**：分类排行按净额；原支出可下钻看退款拆分。
- **筛选**：现有「报销状态（全部/待报销/已退）」维度扩展含「待退款」。
- **组件复用**：走 `docs/dev/components.md` 既有弹窗/图标/数字键盘，不新造变体。

> 📝 **交互批注（任意补充）**：「你来决定就好，我的要求就是：好看、易用、流畅、美观、明确。」

## 10. 测试计划（一期重点）

- 单笔（已到账）退款：净额、原账户余额、到账账户余额。
- 多笔部分退款：累计净额、floor 0。
- 跨账户退款：原账户扣全额、到账账户加退款。
- 全额退款：净额 = 0。
- **待到账退款**：不进余额、不进净额、不进收支统计，只在「待退款」清单出现。
- **待到账 → 标记已到账**：此时余额/净额才变化，条目落到时间线到账日。
- **超额拦截**：Σ退款欲超过原额 → 截到剩余可退 + 提示。
- 删除原支出 → 退款级联删除；删除退款 → 净额恢复。
- 老数据迁移：`refundedAmount>0` → 合成已到账退款条目后净额/余额不变。
- 备份 round-trip：含退款（含待到账）的数据导出再导入一致。
- 收支统计：退款不计收入、不重复计支出。
