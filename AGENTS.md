# Repository Guidelines

## 项目结构与模块组织
本项目是 Flutter 应用，包名为 `verifin`。主要 Dart 源码位于 `lib/`：入口文件 `lib/main.dart` 只保留应用入口与根组件，页面按功能拆分在 `lib/pages/`（首页、预算、交易、资产、看板、我的、记账详情、底部导航壳、共享弹窗），领域层在 `lib/app/`（模型、`VeriFinController` 状态控制、主题、工具函数、通用组件、AI 对话记账 `ai/`、截图识账端上 OCR `screenshot_recognizer_*`；`app_version.dart` 由发布脚本更新版本号），SQLite 数据层在 `lib/data/`（`AppDatabase` 建表迁移、`LedgerRepository` 接口 + `SqliteLedgerRepository` 账目类整表读写、`database_factory*` 数据库工厂），偏好类键值存储在 `lib/local_storage/`。测试代码位于 `test/`，按领域拆分（`navigation_settings`/`accounts_assets`/`entries`/`budget`/`panels`/`backup`/`controller_unit`/`pure` 等 `*_test.dart`），共享脚手架与内存仓储在 `test/support/`，真实 SQLite 覆盖在 `test/repository_test.dart` 与 `test/controller_persistence_test.dart`。Android 平台工程位于 `android/`。依赖与项目配置集中在 `pubspec.yaml`、`pubspec.lock` 和 `analysis_options.yaml`。

## 开发、测试与预览命令
- `flutter pub get`：安装或刷新 Flutter/Dart 依赖。
- `flutter run -d <android-device-id> --flavor github`：在 Android 模拟器或真机上本地预览和调试（项目定义了 `github`/`play` flavor，本地 Android 构建/运行必须带 `--flavor github`）。
- `flutter analyze`：执行静态检查，使用 `flutter_lints` 规则。
- `flutter test`：运行 `test/` 下的全部测试。
- `scripts/publish.sh patch`：发布补丁版本，脚本会更新版本号、提交、创建 `vX.Y.Z` 标签并推送。Windows 上用等价的 `scripts/publish.ps1 patch`（PowerShell），两脚本逻辑一致，须同步维护。

Android 安装包不在本机打包；最终 APK/AAB 由 GitHub CI 负责生成。普通提交不触发 Actions，只有推送 `vX.Y.Z` 标签时才构建。CI 出两种包：**GitHub 自分发的 arm64-v8a 单架构 APK（`--flavor github`）** 与 **Google Play 用的 AAB（`--flavor play --dart-define=SELF_UPDATE=false`）**，均开启 R8 代码/资源裁剪（体积约减半；反射依赖点靠 `proguard-rules.pro` keep 保护，新增反射依赖须补规则），默认发布 GitHub **预发布**（不标记 Latest）；真机验收通过后，在 GitHub 手动把该 release 改为正式版（设为 Latest）。不要在常规开发流程中运行或依赖本机 `flutter build apk` 作为交付依据。**分发渠道 flavor**：应用内自更新只用于 github flavor；play flavor 移除 `REQUEST_INSTALL_PACKAGES`（`android/app/src/play/AndroidManifest.xml`）并隐藏自更新入口（`kSelfUpdateEnabled`），以遵从 Play 政策。

**发版必须经用户明确同意**：只有用户明说「发布 patch/minor/major 版本」之类时才发版，绝不擅自发版（打标签触发 CI、不可逆）。发版前先把 `CHANGELOG.md` 顶部的 `## [Unreleased]` 段改名为本次版本号加日期，并在其上新开一个空的 `## [Unreleased]`，随发版提交一起提交，再跑 `scripts/publish.sh`。

## 代码风格与命名规范
遵循 Dart 默认格式和 `analysis_options.yaml` 中启用的 `flutter_lints`。提交前使用 `dart format .` 格式化代码。使用两个空格缩进。Dart 文件使用 `snake_case.dart`，类和 Widget 使用 `UpperCamelCase`，私有成员使用前导下划线。优先保持 Widget 简洁，只有出现明确复用或复杂度上升时再抽取组件。

## 代码规范 · 组件化与工程约定
面向「商业级、可长期维护」的硬约定。新增或修改代码时必须遵守。

### 组件化与复用
- **写任何 widget / 弹窗 / 对话框 / 格式化 / 计算之前，先查 `docs/dev/components.md` 组件清单**。命中现成件就复用，或给它加参数扩展；**不要新建变体、不要复制粘贴脚手架**。
- 同一 UI 片段或逻辑在 **≥2 个文件**出现，即抽为共享件（通用组件→`common_widgets.dart`，弹窗 helper→`sheets.dart`，记账相关 widget→`entry_sheets.dart`，纯计算→对应 `*_math`/`*_tree` 模块），差异用参数表达。抽取后同步更新组件清单。
- **图标渲染唯一入口**：分类图标一律 `CategoryIconBox` / `CategoryGlyph`，账户图标一律 `AccountIconBox`；**禁止在渲染点直接调 `iconForCode`**（否则 emoji 图标会回退成钱包图标）。
- **指标小块**：Row 里「居中列 label + value[ + detail]」的标准指标用 `SummaryMetric`，不要复制一个只有文案不同的同构变体。确需不同呈现（带色块背景 / 图标 / 横向排名行 / 明显不同字号）时可建局部 widget，但先确认无法用 `SummaryMetric` 的参数覆盖——不为「凑统一」把它塞成配置怪物（过度抽象同样是债）。

### 弹窗与对话框
- 底部弹窗对外暴露顶层 `show<名>Sheet(context, ...)` 函数（内部封 `showModalBottomSheet` + 统一 chrome），**不要在页面里裸包 `showModalBottomSheet`**。确认框用 `showConfirmDialog`（`destructive:true` 走红色），文本输入用 `showTextInputDialog`，单选用 `showOptionSheet`，金额用数字键盘，分类用分类选择器，账户用 `showAccountPickerSheet`。**禁止内联复制 `AlertDialog` / `showModalBottomSheet` 脚手架**。
- helper 第一参数统一具名 `context:`；取消 / 未选一律返回 `null`；「特殊选项」哨兵集中定义并注释（如 `categoryPickerAll`），调用方用命名常量判别，不散写字面量。
- 需要触感（`hapticsEnabled`）的组件由 helper 内部从 `VeriFinScope` 取，调用方不手传。

### 颜色 / 格式化 / 设计令牌
- 金额上色只用 `colorForType` / `accountBalanceColor`；金额文本只用 `formatAmount` / `formatSignedAmount` / `formatCompactAmount`；日期 / 月份只用 l10n 的 `dateMonthDay` / `yearMonth`。**禁止内联手拼**金额或日期字符串。
- 颜色用 `veri*` 常量、圆角用 `veriRadius*`、主色 `#346edb`（`veriRoyal`）。禁止裸写 `Color(0x...)` 或魔法圆角；局部确需的特例（如键盘配色）须注释理由。

### 错误处理与日志
- `catch` 到异常后：**要么** `logger.error(msg, source: '<域>', error: e)` 记日志 **并**给用户可见反馈（SnackBar / 对话框），**要么**是有意的降级（须注释说明为何可静默）。禁止无注释的空 `catch (_) {}` 吞异常。backup / import / ai / webdav / 数据库等易错路径的 catch 必须写日志。
- 日志走 `lib/app/logging` 的 `AppLogger`（分级、可导出、隐私友好），不要用 `print` / `debugPrint` 作为生产日志。

### 异步与状态
- 任何 `await` 之后再用 `BuildContext`（`Navigator` / `ScaffoldMessenger` / `Theme.of` 等）前，必须先 `if (!mounted) return;`（`State`）或 `if (!context.mounted) return;`（无 State）。
- 应用状态一律经 `VeriFinController` 读写（`VeriFinScope.of(context)`），不要在页面里另存权威状态；持久化经 controller 的 `_persistX`，不要在 UI 层直接碰 repository / KV。

### 文案本地化
- 新增用户可见文案必须写入 `lib/l10n/app_zh.arb`（模板）+ `app_en.arb`（同步），经 `AppLocalizations.of(context)` 引用，**绝不硬编码中文**（豁免仅限：法律正文、品牌/银行图标名、CSV 模板表头与逐行导入错误、无 context 的网络/桥接错误）。

## 测试规范
使用 `flutter_test` 编写 Widget 测试和基础行为测试。测试文件放在 `test/` 下，并以 `_test.dart` 结尾。测试应覆盖用户可见行为，例如文本渲染、按钮交互、导航和表单校验。修改功能后至少运行 `flutter analyze` 和 `flutter test`。

## 提交与 Pull Request 规范
提交信息使用简短明确的 `type: summary` 格式，例如 `feat: add login screen`、`fix: correct counter test`。提交代码时不要包含任何 AI 或 Codex 署名信息，包括 `Generated by ...`、`Co-authored-by: ...` 等。PR 应包含变更说明、测试结果、关联 issue（如有）以及 UI 变更截图或录屏。

## 文档同步要求
每次修改代码、功能、配置或流程时，都要检查 `docs/`、`AGENTS.md` 和 `README.md` 是否需要同步更新。若变更影响开发命令、架构说明、交付流程、配置方式或用户可见行为，必须在同一次变更中更新相关文档。若 `docs/` 目录暂不存在或无相关文件，也应确认 `README.md` 与本文件是否需要补充。

凡用户可见的改动（新功能、界面/交互变化、修复、隐私安全项等），在同一次变更中往 `CHANGELOG.md` 的 `## [Unreleased]` 段落加一行（分类到 新增/优化/修复/安全 等小节）；纯内部重构、改测试/文档等对用户无感的不写。发版时把 `Unreleased` 提升为版本号（见「开发、测试与预览命令」的发版说明）。

## Agent 工作约定
与用户沟通时使用中文。改动应聚焦在用户请求范围内，不要重写生成的平台工程文件，除非任务明确要求。优先沿用 Flutter 官方约定和当前项目配置，不要为简单需求引入额外工具链。

新增用户可见文案必须写入 `lib/l10n/` 的 ARB 文件（`app_zh.arb` 模板 + `app_en.arb` 同步），通过 `AppLocalizations.of(context)` 引用，绝不硬编码中文。应用已支持中英双语（设置页「语言」切换，存量迁移 2026-07 完成），真机验证清单见 `docs/dev/i18n-verification.md`。关键技术决策记录在 `docs/dev/tech-decisions.md`。

凡是涉及本地存储、图片、文件、安装包或系统入口的功能，都必须确认 Android 端有真实实现，并同步检查持久化、权限、签名和进程重启后的行为。
